# Аналитические запросы и индексы — продолжение курсовой по базам данных

> Командный проект, университетский курс по базам данных  
> Продолжение работы по проектированию БД больницы: написание аналитических SQL-запросов и оптимизация через индексы

---

## О проекте

Вторая часть курсовой работы по базам данных. На основе спроектированной ранее схемы больницы написали три аналитических запроса разной сложности и добавили индексы для их оптимизации. Каждый запрос решает реальную бизнес-задачу: найти наиболее загруженных врачей, выявить ценных пациентов, отследить возвращающихся клиентов.

## Запросы

### Запрос 1 — Загруженные врачи

Находит врачей, у которых больше 20 приёмов. Для каждого показывает общее число приёмов, а также сколько из них завершились успешно и сколько было отменено.

Технически: `JOIN` с таблицами `DOCTOR` и `DEPARTMENT`, условное агрегирование через `CASE WHEN` внутри `COUNT` — это позволяет посчитать завершённые и отменённые приёмы в одном проходе без подзапросов. Фильтрация через `HAVING` после группировки. Статус приводится к нижнему регистру через `LOWER()`, чтобы не зависеть от регистра в данных.

Такой запрос полезен для оценки нагрузки на персонал и анализа качества записи — высокий процент отмен у конкретного врача может сигнализировать о проблемах с расписанием.

---

### Запрос 2 — Ценные пациенты

Находит пациентов, которые одновременно посетили минимум три разных отдела и потратили больше среднего среди всех платящих пациентов.

Технически: два CTE (`WITH`). Первый — `patient_stats` — агрегирует по каждому пациенту количество уникальных отделов, приёмов, диагнозов и суммарные траты. `COALESCE` используется для замены `NULL` нулём там, где у пациента не было процедур. Второй CTE — `avg_spent` — считает среднее только среди тех, кто вообще что-то тратил, чтобы не занижать среднее за счёт нулей. Основной запрос фильтрует по обоим условиям одновременно.

`LEFT JOIN` на `DIAGNOSIS` и `APPOINTMENT_PROCEDURE` — намеренное решение: пациент попадёт в выборку даже если у него нет диагноза или процедур, что соответствует реальной ситуации.

---

### Запрос 3 — Возвращающиеся пациенты

Находит приёмы, на которые пациент пришёл после минимум трёх предыдущих визитов. Показывает, сколько именно ранних приёмов было у пациента до этого.

Технически: self-join таблицы `APPOINTMENT` на саму себя. Алиас `a2` — текущий приём, `a1` — более ранние приёмы того же пациента (условие `a1.Date < a2.Date` и совпадение `Patient_ID`). После группировки `HAVING COUNT(a1.ID) >= 3` оставляет только те приёмы, перед которыми было хотя бы три других. Это позволяет отследить действительно лояльных пациентов, которые возвращаются в клинику регулярно.

## Индексы

Четыре индекса, которые ускоряют выполнение запросов выше.

**Составной индекс на `APPOINTMENT`** по полям `(Patient_ID, Doctor_ID, Status, Date, Time)` — самый важный. Покрывает сразу несколько сценариев: фильтрацию по статусу (запрос 1), поиск приёмов конкретного пациента с сортировкой по дате (запрос 3), джойны по пациенту и врачу (запрос 2). Составной индекс работает эффективнее нескольких отдельных, если запросы используют эти поля вместе.

**Индекс на `DOCTOR(Department_ID)`** — ускоряет `JOIN` между `DOCTOR` и `DEPARTMENT`, который встречается во всех трёх запросах.

**Составной индекс на `APPOINTMENT_PROCEDURE`** по `(Appointment_ID, Procedure_ID)` — оптимизирует работу с таблицей-связкой при подсчёте трат пациента в запросе 2.

**Индекс на `DIAGNOSIS(Appointment_ID)`** — ускоряет `LEFT JOIN` с `DIAGNOSIS` в запросе 2.

## Выводы

Эта часть курсовой оказалась интереснее первой в том смысле, что нужно было не просто написать рабочий запрос, а думать о том как он выполняется внутри. Self-join в третьем запросе — хороший пример задачи, которую сложно решить без понимания того как база обрабатывает соединение таблицы с собой. CTE во втором запросе сделали код читаемым: вместо одного большого запроса с вложенными подзапросами — два именованных шага с понятной логикой.

Индексы добавляли уже после запросов, анализируя какие поля участвуют в условиях и соединениях чаще всего. Это помогло понять, что индекс — это не просто «ускоритель», а решение под конкретный паттерн использования данных.
